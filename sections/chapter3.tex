\chapter{系统需求分析}

\section{系统建设目标}
本文系统面向人工智能专业的医学图像处理实验。系统目标是构建一个 MRI 医学图像生成平台，并贯通数据预处理、StyleGAN2 二维生成、连续切片组织、点云初始化、3DGS 训练结果展示和 AI 辅助说明。与命令行脚本相比，平台更强调流程清楚、结果可见和实验可复现。

谁会使用这个平台？主要用户包括进行医学图像生成实验的学生、学习 StyleGAN2 与 3DGS 的学习者，以及需要展示生成式医学影像流程的教学人员。平台不面向临床诊断，也不输出医学诊断结论。平台定位是算法教学、毕业设计展示和工程实验验证。

\section{功能需求分析}
系统功能围绕完整实验流程展开。主要功能包括数据预处理、模型训练、切片生成、连续序列生成、点云初始化、3DGS 结果管理和 AI 辅助问答。

数据预处理功能需要读取 NIfTI 格式 MRI 文件。系统应自动提取轴向切片，并完成灰度归一化、尺寸缩放、质量过滤和 PNG 保存。系统还需要调用 StyleGAN2 数据集工具生成训练压缩包。为了让用户判断处理是否成功，平台应返回处理日志，并说明输入路径、输出路径、目标尺寸、有效切片数量和异常文件信息。

模型训练功能需要基于预处理数据启动 StyleGAN2-ADA 训练。用户应能设置学习率、批大小和训练迭代量等基础参数。训练任务应在后台运行，因为 StyleGAN2 训练耗时较长。系统应保存模型快照、配置文件和训练日志，以支持论文复核实验材料。

切片生成功能需要加载训练得到的 \texttt{.pkl} 权重文件。系统应根据随机种子、生成数量和截断参数输出二维 MRI 切片。生成结果应保存到结果目录。平台也应直接显示生成图像。该功能用于论文插图整理和答辩演示。

连续序列生成功能需要在两个随机种子对应的潜变量之间插值。系统通过插值得到连续变化的切片序列，并把独立二维样本组织为近似体数据结构的序列。该序列为点云初始化和 3DGS 训练提供输入。

点云初始化功能需要读取连续切片目录。系统将非背景像素映射为空间点，并把灰度值转换为颜色信息，最终输出 PLY 格式点云。该模块是二维生成结果进入三维表达流程的接口。

3DGS 结果管理功能需要归档训练日志、相机参数、checkpoint、渲染图像和训练后点云。平台应通过路径、日志和图像预览展示三维训练结果。这样，三维部分不只是产生文件，还可以形成可解释的实验材料。

AI 辅助问答功能用于解释 MRI 图像生成、StyleGAN2 参数、3DGS 基本概念和平台操作流程。该功能服务教学演示。它不参与核心训练和推理。因此，即使外部模型接口不可用，平台主要功能仍应保持可运行。

\section{性能与可用性需求分析}
系统性能需求主要包括长任务调度和高频操作响应。MRI 批量预处理、StyleGAN2 训练和 3DGS 训练属于计算密集型任务。系统应通过后台进程执行这些任务，避免阻塞 Gradio 界面。切片生成、图像预览和点云初始化属于常用演示操作，系统应尽量在可接受时间内给出日志或输出路径。

系统可用性需求主要包括低门槛操作和清晰反馈。用户不应直接面对复杂命令行参数。用户应通过输入框、滑块、按钮和日志区完成操作。对于数据路径不存在、模型权重缺失或生成数量异常等问题，平台需要返回明确提示。对于正常运行结果，平台应展示图像画廊、输出路径和关键日志。

系统扩展性需求主要体现在模型、三维重建和评价指标三个方向。二维生成部分应支持替换不同 StyleGAN2 权重或不同训练数据。三维部分应能从点云初始化扩展到更完整的 3DGS 训练与渲染。评价部分可以继续加入 FID、SSIM、PSNR、专家评价或下游任务指标。平台代码应保持模块化，避免新增功能时重写整体界面。

\section{可行性分析}
从技术可行性看，本文所用技术都有较成熟的开源实现或 Python 生态支持。StyleGAN2-ADA 有可复用的 PyTorch 工程，NIfTI 数据可由 nibabel 读取，图像处理可使用 NumPy 和 PIL 完成，Gradio 能快速构建交互界面。当前项目已经形成 StyleGAN2 权重、生成切片、初始点云、3DGS 训练日志和渲染结果。这些产物说明核心流程具备实现基础。

从数据可行性看，IXI 数据集公开提供多种脑部 MRI 数据。数据类型包括 T1、T2、PD、MRA 和 DTI 等。数据以 NIfTI 格式发布，适合教学和算法实验\cite{ixiDataset}。本文使用 T1 脑部 MRI 作为主要实验数据。该数据能够支撑二维切片生成和三维序列组织。

从工程可行性看，系统采用“算法脚本 + Gradio 平台”的组织方式。底层脚本负责计算任务，界面层负责参数输入、任务调度和结果展示。两者通过文件路径、日志和子进程调用连接。该方案实现成本适中，也便于调试和展示。

从应用可行性看，平台定位为教学与研究演示系统。平台不是临床诊断软件。因此，平台不承担医疗器械级别的验证要求。论文对生成结果的描述以算法展示、工程闭环和可视化验证为主。论文不把平台输出解释为临床可用数据。
